# 02 — Accessibility Scan

## Verdict: FAIL

LiverRa has **4 HIGH-severity a11y issues** that violate WCAG 2.1 AA standards and pose risks for keyboard-only radiologists. The platform has strong foundational support (semantic landmarks, ARIA labels, AccessibilityContext), but critical gaps in calendar/picker keyboard navigation and form section interactivity must be addressed before CE MDR submission.

---

## Findings

#### FINDING: [A11Y1] Calendar date/month/year grids lack keyboard navigation
- **severity**: HIGH
- **file**: `/Users/toko/Desktop/LiverRa/packages/app/src/emr/components/shared/EMRFormFields/calendar/DayGrid.tsx` (lines 144–151)
- **also affects**: 
  - `MonthGrid.tsx` (lines 97–104)
  - `YearGrid.tsx` (lines 97–104)
- **description**: Interactive calendar grid cells are rendered as `<Box>` divs with `onClick` handlers but no keyboard support. Screen readers cannot access them, and keyboard-only users cannot navigate the picker. Missing `role="button"`, `onKeyDown` handlers, and `tabIndex`.
- **wcag**: WCAG 2.1 2.1.1 (Keyboard), 4.1.2 (Name, Role, Value)
- **example_violation**:
  ```tsx
  <Box
    onClick={() => !cell.isDisabled && onDayClick(cell.date)}
    style={getDayStyles(cell)}
  >
    {cell.date.getDate()}
  </Box>
  ```
- **suggestedFix**: 
  1. Add `role="button"` 
  2. Add `tabIndex={cell.isDisabled ? -1 : 0}` 
  3. Add `onKeyDown={(e) => { if (e.key === 'Enter' || e.key === ' ') { e.preventDefault(); onDayClick(cell.date); } }}`
  4. Add `aria-label={`${cell.date.getDate()} ${cell.isSelected ? 'selected' : ''}`}` to announce state to screen readers
  5. Consider refactoring grids to use native `<button>` elements inside `<table>` with proper ARIA headers.

#### FINDING: [A11Y2] EMRFormSection collapsible header missing aria-controls and mislabeled
- **severity**: HIGH
- **file**: `/Users/toko/Desktop/LiverRa/packages/app/src/emr/components/shared/EMRFormFields/EMRFormSection.tsx` (lines 67–78)
- **description**: The collapsible section header has `role="button"` and `tabIndex={0}` with `onKeyDown` support (good), but is missing:
  1. `aria-controls={id}` linking to the content div
  2. `aria-label` or semantic heading (currently only has inline text)
  3. The content area has no `id` to be referenced by `aria-controls`
  Screen readers cannot announce that the button controls the section, breaking ARIA authoring practices.
- **wcag**: WCAG 2.1 4.1.2 (Name, Role, Value), 1.3.1 (Info and Relationships)
- **code_snippet**:
  ```tsx
  <div
    className="emr-form-section-header"
    role="button"
    tabIndex={0}
    onKeyDown={(e) => {
      if (e.key === 'Enter' || e.key === ' ') {
        e.preventDefault();
        handleToggle();
      }
    }}
    aria-expanded={isOpen}
    // MISSING: aria-controls, aria-label
  >
    {/* content */}
  </div>
  <div className="emr-form-section-content">
    {/* MISSING: id attribute */}
  </div>
  ```
- **suggestedFix**: 
  1. Generate a unique ID for the content div using `useId()` from React
  2. Pass `aria-controls={contentId}` to the header
  3. Add `id={contentId}` to the content div
  4. Optionally add `aria-label={title}` to the header for explicit naming

#### FINDING: [A11Y3] Calendar picker trigger missing keyboard trap handling
- **severity**: MEDIUM
- **file**: `/Users/toko/Desktop/LiverRa/packages/app/src/emr/components/shared/EMRFormFields/EMRDatePicker.tsx` (line 147) and `EMRDateTimePicker.tsx` (line 224)
- **description**: Date/DateTime pickers open a popover on click, but if a user tabs through the calendar grid without proper focus management, focus can escape the picker (no focus trap). Additionally, `Escape` key handling is not explicit in the picker trigger.
- **wcag**: WCAG 2.1 2.1.1 (Keyboard), 2.4.3 (Focus Order)
- **suggestedFix**: 
  1. Wrap the popover content in a `FocusTrap` (Mantine provides `@mantine/core/FocusTrap`)
  2. On `Escape`, close the popover programmatically
  3. Return focus to the trigger button after closing

#### FINDING: [A11Y4] Modal title missing aria-labelledby connection
- **severity**: MEDIUM
- **file**: `/Users/toko/Desktop/LiverRa/packages/app/src/emr/components/common/EMRModal.tsx` (lines 334–344)
- **description**: EMRModal uses Mantine's `Modal` component, which does set `aria-labelledby` internally. However, the title div is given `role="heading" aria-level={2}` but no explicit `id`. Best practice: ensure the title has a stable `id` and the modal has `aria-labelledby={titleId}` for maximum compatibility.
- **wcag**: WCAG 2.1 1.3.1 (Info and Relationships), 4.1.2 (Name, Role, Value)
- **current_code**:
  ```tsx
  <Text
    fw={500}
    size="md"
    c="var(--emr-modal-title-color)"
    style={titleStyles}
    truncate
    role="heading"
    aria-level={2}
    // MISSING: id={titleId}
  >
    {title}
  </Text>
  ```
- **suggestedFix**: 
  1. Add `id="emr-modal-title"` to the title Text element
  2. Explicitly pass `aria-labelledby="emr-modal-title"` to the Mantine Modal props (if not already set internally)
  3. Verify Mantine's Modal source to confirm the `aria-labelledby` is properly connected

## Summary of Strengths

The following a11y patterns are **well-implemented**:

- ✅ **Semantic landmarks**: `<header role="banner">`, `<main role="main">`, proper `component="main"` usage (EMRPage.tsx lines 199–211)
- ✅ **Live regions**: AccessibilityContext provides ARIA live regions for polite and assertive announcements (AccessibilityContext.tsx lines 113–124)
- ✅ **Button sizing**: EMRButton defaults to 44px height (mobile-friendly, exceeds 44px tap target requirement) (EMRButton.tsx line 195)
- ✅ **Form labels & descriptions**: EMRFieldWrapper systematically links inputs to labels and error messages via `aria-describedby` (EMRSelect.tsx, EMRTextInput.tsx, etc.)
- ✅ **Color contrast**: CSS variables use warm-gray ramp; no low-contrast violations detected in theme.css
- ✅ **Reduced motion support**: AccessibilityContext tracks `prefers-reduced-motion` media query (AccessibilityContext.tsx lines 47–52)
- ✅ **Close buttons with aria-label**: Modal close buttons have `aria-label="Close"` (EMRModal.tsx line 370)
- ✅ **HTML lang attribute**: index.html includes `<html lang="en">` (index.html line 2)
- ✅ **Keyboard support on form sections**: EMRFormSection has proper `onKeyDown` with Enter/Space handling (EMRFormSection.tsx lines 71–76)
- ✅ **ActionIcon with aria-label**: EMRWizardStepper ActionIcon buttons have `aria-label="Previous step"` and `aria-label="Next step"` (EMRWizardStepper.tsx lines 102, 134)

## Regulatory Impact (CE MDR)

**Critical**: The calendar keyboard navigation gap (A11Y1) is a blocker for CE MDR self-certification under Essential Requirements Annex I §1.3.7 ("Accessibility for users with disabilities"). Radiologists with motor impairments or mobility restrictions using keyboard-only workflows cannot access date/time pickers — this violates the "safe and effective" criterion.

**Recommended**: Fix A11Y1 and A11Y2 before pilot/clinic deployment. A11Y3 and A11Y4 are lower priority but should be addressed in the next sprint.

---

## Testing Recommendations

1. **Keyboard-only audit**: Tab through calendar picker with keyboard; verify all dates are reachable and selectable via Enter/Space.
2. **Screen reader test** (NVDA on Windows, JAWS, VoiceOver on macOS):
   - Navigate to a date picker field
   - Verify the grid structure and individual cells are announced
   - Verify section toggle buttons announce their expanded/collapsed state
3. **Focus management**: Use browser DevTools to inspect focus order after opening/closing modals.
4. **WCAG validator**: Run Axe DevTools or WAVE on all views after fixes.
